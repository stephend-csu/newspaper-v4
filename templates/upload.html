Address Processing Pipeline & APIs

Step 1 — Data Extraction
PDF Address & Newspaper Parser
Parses layout text from the uploaded PDF. Extracts house numbers, street headers, and newspaper codes (EBT, WSJ, SFC, NYT, CAP, UST, WLD, STD, LAT, FT, IBD, BAR) while discarding route metadata codes like 7D or 1.
Technology: Python pypdf + Regular Expressions
Step 2 — Address Validation & Classification
County Address Classifier
Formats raw extracted street lines without city names and validates them against official Contra Costa County city boundaries (Walnut Creek, Concord, Pleasant Hill, Lafayette, Danville, San Ramon, etc.). Discovers true cities via ArcGIS geocoding without default fallbacks.
Module: geocoder.py (validate_and_classify_addresses)
Step 3 — Geocoding
ArcGIS World Geocoding Service API
Queries the ArcGIS World Geocode REST API to determine exact latitude and longitude geographic coordinates for every valid delivery address. Operates live without hardcoded coordinate fallbacks to ensure precise pin placements.
API: geocode.arcgis.com/arcgis/rest/services/World/GeocodeServer
Step 4 — Route Optimization & Street Clustering
OSRM API & Street-Clustering Engine
Computes real-world road network driving distances starting from 2505 Dean Lesher Dr, Concord, CA (including mandatory stop 923 Pacific Ct). Solves the Traveling Salesperson Problem (TSP), calculates leg miles between stops, and clusters multi-address streets into consecutive ascending or descending house-number blocks.
API: router.project-osrm.org + Local 2-Opt TSP & Street-Clustering Engine
Step 5 — Repository Deployment
GitHub REST API Sync
Commits the newly generated Chapters.csv route dataset and metadata.json directly to the GitHub repository (main branch) in a single atomic commit via Git Trees API, triggering a clean GitHub Pages build with a PST upload timestamp.
API: api.github.com/repos/stephend-csu/newspaper-v4/git/trees
